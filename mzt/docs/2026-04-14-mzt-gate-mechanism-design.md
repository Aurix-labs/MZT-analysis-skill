# MZT Gate 机制与问题类型判断设计

## 概述

**问题背景**：当前 MZT skill 存在"Agent 未追问用户情况就输出完整分析"的问题，违背了"具体问题具体分析"的核心原则。

**解决方案**：增加"问题类型判断"阶段和"Gate 机制"，强制 Agent 在关键节点等待用户确认。

**核心变化**：

```
原流程：
问题建构 → 方法论选择 → 分析输出 → 验证迭代

新流程：
问题类型判断 → [Gate] → 问题建构 → [Gate] → 方法论选择 → [Gate] → 分析输出 → 验证迭代
```

---

## 一、阶段 0：问题类型判断（新增）

### 1.1 问题类型分类

| 类型 | 定义 | 典型示例 | 追问策略 |
|------|------|----------|----------|
| **实践型** | 涉及用户个人决策、资源、处境 | "我创业应该选择什么方向" | 必须追问 |
| **认识型** | 探究客观事实、历史事件、普遍规律 | "秦始皇为什么统一六国" | 直接分析 |
| **准认识型** | 表面通用，实则可个性化 | "AI时代程序员如何规划职业" | 可直接回答，提示个性化 |
| **边界模糊型** | 无法从问题本身判断 | "中国芯片产业发展前景" | 一轮追问确定语境 |

### 1.2 判断流程

```
输入问题
    │
    ▼
第一判断：是否包含第一人称主体？
关键词："我/我的/我们/我应该/我要"
    │
    ├─ 是 → 实践型问题 → 必须追问 → 进入问题建构
    │
    └─ 否 → 第二判断：是否涉及实践决策？
            关键词："应该/如何选择/怎么办/如何做"
            │
            ├─ 否 → 认识型问题 → 直接分析
            │
            └─ 是 → 第三判断：是否明显涉及普遍规律？
                    │
                    ├─ 是 → 准认识型问题 → 可直接回答
                    │
                    └─ 否 → 边界模糊型问题 → 一轮追问
```

### 1.3 追问话术模板

**边界模糊型问题通用模板**：

```
在开始分析之前，我需要确认一下：您问这个问题是希望[选项A]，还是[选项B]？

- [选项A]：[说明A]
- [选项B]：[说明B]

您也可以说'直接分析'，我会给出通用角度的分析。
```

**各领域追问话术**：

| 问题领域 | 追问话术 |
|----------|----------|
| 行业趋势类 | "您问这个问题是出于什么考虑？是研究需要、投资决策，还是个人职业规划？" |
| 技术方向类 | "您是希望了解一般性的技术趋势，还是需要针对您的情况给出建议？" |
| 管理方法类 | "您是在学习管理知识，还是在解决具体的团队问题？" |
| 政策影响类 | "您是从业者、研究者，还是政策关注者？不同角度分析重点不同。" |

## 二、Gate 机制

### 2.1 Gate 定义

**Gate** = 阶段间的强制等待点，Agent 必须收到用户确认后才能进入下一阶段。

### 2.2 Gate 总览

| 阶段 | Gate 名称 | 触发条件 | Agent 必须做的事 | 跳过条件 |
|------|-----------|----------|------------------|----------|
| 阶段 0 | 类型判断 Gate | 实践型或边界模糊型问题 | 追问并等待 | 用户说"直接分析" |
| 阶段 1 | 问题确认 Gate | 所有情况 | 复述问题并等待确认 | 用户说"直接分析" |
| 阶段 2 | 方向确认 Gate | 所有情况 | 输出方法论选择并等待确认 | 用户说"直接分析" |
| 阶段 3 | 报告完成 Gate | 报告输出后 | 等待用户评价 | 用户说"结束" |

### 2.3 Gate 执行规范

**Agent 必须遵守的规则**：

```
1. 输出追问或确认请求后，立即停止输出
2. 不得在同一消息中继续输出后续阶段内容
3. 必须等待用户下一条消息才能继续
4. 用户说"直接分析"或"跳过"时，可直接进入下一阶段
```

**禁止的行为**：

```
❌ 追问后紧接着输出分析框架
❌ 自行判断"问题已清晰"并跳过 Gate
❌ 将多个阶段的追问合并为一条消息
❌ 假设用户会回答，预设后续内容
❌ 输出完整报告后才询问"是否需要调整"
```

### 2.4 跳过机制

用户可在任何 Gate 说以下任一词语跳过等待：

- "直接分析"
- "跳过"
- "继续"
- "不用问了"

Agent 收到跳过指令后，基于已有信息或一般性假设继续进入下一阶段。

## 三、Gate 提示语设计

### 3.1 Gate 0：类型判断 Gate

**触发条件**：问题类型为实践型或边界模糊型

**Agent 输出格式**：

```
在开始分析之前，我需要了解：[追问内容]

您可以回答上述问题，或说'直接分析'跳过追问。
```

### 3.2 Gate 1：问题确认 Gate

**触发条件**：所有情况（阶段 0 判断后进入此阶段）

**Agent 输出格式**：

```
让我复述您的问题核心：[复述内容]

这样理解准确吗？您可以说'问题已理解'确认，或补充更多信息。
```

### 3.3 Gate 2：方向确认 Gate

**触发条件**：用户确认问题理解后

**Agent 输出格式**：

```
基于以上分析方向，可能得出[简要预期结论]。

这是您期待的方向吗？请确认'方向正确'，或提出调整意见。
```

### 3.4 Gate 3：报告完成 Gate

**触发条件**：报告输出完成后

**Agent 输出格式**：

```
这次分析中，最让您有启发的观点是什么？还有哪些疑问没有解决？
```

## 四、阶段 1 追问话术设计

### 4.1 追问维度

| 追问维度 | 适用场景 | 追问话术 |
|----------|----------|----------|
| **背景现状** | 不了解用户起点 | "您目前处于什么阶段/位置？" |
| **资源禀赋** | 涉及资源分配决策 | "您目前掌握的关键资源有哪些？（技术、人脉、资金、时间）" |
| **目标定义** | 成功标准不明确 | "您心目中'成功'的定义是什么？[时间]内达到什么状态算成功？" |
| **核心约束** | 存在明显限制 | "您面临的最大约束是什么？是[约束A]还是[约束B]？" |
| **风险偏好** | 涉及不确定性选择 | "您对风险的态度是偏向稳健还是激进？" |

### 4.2 追问原则

1. **目的是发现矛盾，不是填清单**
2. **用户可随时说"直接分析"跳过追问**
3. **Agent 不主动判断"已清晰"，由用户确认**
4. **每次追问后必须等待用户回复，不得继续输出**
5. **一次只问一个问题，不堆砌追问**

### 4.3 追问深度控制

| 判断条件 | 说明 |
|----------|------|
| 核心矛盾已明确 | 能说出问题背后的主要冲突 |
| 关键信息已充足 | 能做出有针对性的分析 |
| 用户表示不耐烦 | 用户说"够了"、"直接分析" |
| 追问超过 3 轮 | 避免过度追问，基于已有信息继续 |

## 五、执行心态调整

### 5.1 从"输出导向"转向"对话导向"

| 旧心态 | 新心态 |
|--------|--------|
| 我要输出一个完整的分析报告 | 我要通过对话帮助用户理清问题 |
| 害怕给出"不完整"的答案 | 追问比完整分析更有价值 |
| 尽快展示专业性 | 先理解，再分析 |
| 用户问什么就答什么 | 用户问的未必是真正需要的 |

### 5.2 核心原则

> "没有调查，没有发言权。"

在了解用户具体情况之前，任何分析都是主观臆断。

### 5.3 判断标准

问自己：**这个分析是针对"这个用户"的，还是针对"任何用户"的？**

- 如果任何用户都能得到同样的回答 → 可能跳过了追问
- 如果回答用到了用户的具体信息 → 说明追问有效

## 六、边界情况处理

### 6.1 用户不耐烦

| 用户表现 | Agent 行为 |
|----------|------------|
| "别问了，直接说" | 立即跳过追问，基于一般假设继续 |
| "怎么这么多问题" | 解释追问目的，询问是否继续 |
| 连续两次"直接分析" | 不再追问，直接输出分析 |

### 6.2 用户回答模糊

| 用户回答 | Agent 行为 |
|----------|------------|
| "都可以" | 选择最常见的情况作为假设，说明假设 |
| "不知道" | 提供选项，降低回答难度 |
| 答非所问 | 换一种方式追问，或基于已有信息继续 |

### 6.3 问题类型误判

| 情况 | Agent 行为 |
|------|------------|
| 判断为认识型，但用户追问涉及个人情况 | 重新判断问题类型，补充追问 |
| 判断为实践型，但用户说"我只是想了解" | 调整为认识型处理，直接分析 |

## 七、完整流程图

```
阶段 0: 问题类型判断（新增）
    │
    ├── 判断问题性质（实践型/认识型/边界模糊型）
    ├── 若需追问 → [Gate 0: 等待用户回复]
    └── 若认识型 → 直接进入阶段 2
    │
    ▼
阶段 1: 问题建构（能力一）
    │
    ├── 识别核心矛盾
    ├── 追问缺失信息（围绕矛盾）
    └── [Gate 1: 用户确认"问题已理解"]
    │
    ▼
阶段 2: 方法论选择（能力二）
    │
    ├── 识别问题特征维度
    ├── 选择 2-3 个方法论
    ├── 论证选择理由
    └── [Gate 2: 用户确认"方向正确"]
    │
    ▼
阶段 3: 分析输出（能力三内嵌）
    │
    ├── 输出七章节报告
    └── 包含批判性反思章节
    │
    ▼
阶段 4: 验证迭代（能力四）
    │
    ├── 提出可验证假设
    └── 设计验证方法
    │
    ▼
阶段 5: 用户评价与反馈
    │
    ├── [Gate 3: 用户评价]
    ├── 苏格拉底式确认
    └── 案例记录沉淀
```

## 九、测试用例

| 用户连续两次"直接分析" | 不再追问，直接输出分析 |

## 十、解决的问题

| 问题 | 解决方式 |
|------|----------|
| 没追问就分析 | Gate 0 强制追问实践型/边界模糊型问题 |
| 不等待用户回复 | Gate 机制禁止继续输出 |
| 套公式、不具体分析 | 问题类型判断区分处理策略 |
| 急于展示完整答案 | 执行心态从"输出"转向"对话" |
| 问题类型不区分 | 四种问题类型分类，不同策略 |
