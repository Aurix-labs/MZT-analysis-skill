# 输出模板

各章节的完整输出模板，输出报告/各章节时务必参照。

---

## 方法论选择输出模板

```markdown
## 方法论选择说明

### 识别到的问题特征
- [特征1]：[说明]
- [特征2]：[说明]

### 选择的方法论（共 2-3 个）

| 方法论 | 选择理由 | 将用于分析 |
|--------|----------|------------|
| [方法论] | [基于哪个特征选择] | [解决什么问题] |

### 适用边界
- 本分析适用于：[范围]
- 本分析不适用于：[范围]

### 预期分析方向
基于以上方法论，本分析将：
- [分析重点1]
- [分析重点2]

我将基于此方向直接给出分析。若您想换个方向，告诉我即可调整。
```

## 批判性反思模板（能力三）

```markdown
## 批判性反思

### 核心假设
本分析基于以下假设：
1. [假设1] — 如果这个假设不成立，结论需要修改
2. [假设2] — 验证方法：[如何验证]

### 可能的反驳
- 反驳视角一：[可能的反对意见]
  - 回应：[如何回应]

### 分析局限
- 本分析未能覆盖：[局限]
- 建议补充：[后续可以做什么]
```

## 实践检验模板

```markdown
## 实践检验

### 可证伪假设
基于本分析，可以验证的核心假设：
1. [假设1]：若此假设成立，预期可观察到 [具体现象]；若不成立，将观察到 [相反现象]

### 验证方法建议
| 验证项 | 方法 | 成功标准 | 失败标准 |
|--------|------|----------|----------|
| [验证项] | [方法] | [可观察到的成功信号] | [可观察到的失败信号] |

### Agent 自检：预演反驳
以下是最可能推翻本结论的论据，Agent 在用户验证前主动列出：
1. **反驳1**：[论据] → 强度评估：[高/中/低]，原因：[理由]
2. **反驳2**：[论据] → 强度评估：[高/中/低]，原因：[理由]

> 注：完整的"实践—再认识"闭环需要用户将上述假设带入实际场景验证后返回讨论。本报告提供的是验证脚手架，不是已闭合的结论。
```

## 事实依据章节模板

```markdown
## 事实依据

### 诚实标注

按信息可得性如实标注，禁止编造来源或填占位：

| 情形 | 正确做法 |
|------|----------|
| 有可靠来源 | 标注具体来源（文献、数据出处、检索结果） |
| 来源不确定 | 说明"据……，未经独立验证" |
| 无可靠来源 | **写明信息缺口**，说明哪些信息尚不可知 |
| 需要用户补充 | **向用户追问**，而非用假设填充 |
| 来源不存在 | **禁止编造来源或填占位**，这是最严重的形式主义 |

### 信息缺口

- 哪些关键信息尚未获取？
- 获取方式：可追问用户 / 可检索补充 / 当前不可知（如实说明）
```

## 批判性反思章节模板（报告结构）

```markdown
## 批判性反思

### 核心假设
- 本报告的核心假设是什么？
- 这些假设是否经得起检验？

### 可能的反驳
- 有没有其他可能的解释？
- 反对者会怎么反驳我的结论？

### 局限性
- 本报告最大的局限是什么？
- 什么情况下本报告的结论会失效？

### 改进方向
- 还需要补充哪些信息？
- 分析还可以从哪些角度深化？
```

```markdown
### 问题特征
- 核心矛盾：[描述]
- 问题特征维度：[特征标签]
- 特殊性：[本案例独特之处]

### 方法论应用
- 选择的方法论：[方法论名称]
- 选择理由：[为什么选择]
- 应用效果：[效果描述]

### 判断模式提炼
- 当遇到 [特征] 时，优先考虑 [方法论]
- 注意避免 [常见错误]

### 用户反馈
- 问题解决程度：[是/部分/否]
- 建议可执行性：[是/部分/否]
- 用户补充：[开放回答]
```

### 案例库管理（可选）

> 案例库为跨会话可选沉淀，非核心闭环。用户须主动同意存储后方可执行以下操作。

- **存储位置**：`mzt/cases/` 目录
- **文件命名**：`YYYY-MM-DD-<主题摘要>.md`
- **案例归档**：超过 100 条时，按时间归档到 `cases/archive/`
- **案例索引**：`cases/INDEX.md` 维护案例目录，支持按特征检索
